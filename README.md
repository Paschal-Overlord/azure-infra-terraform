# Azure Production Infrastructure вЂ” Terraform

A complete, production-oriented Azure environment provisioned entirely through Terraform. No manual portal clicks. No hardcoded secrets. Every resource is version-controlled, repeatable, and environment-aware.

This was built to support a real application workload and reflects the kind of infrastructure decisions you'd make when reliability and security actually matter.

---

## What's in here

| File | What it provisions |
|---|---|
| `providers.tf` | AzureRM + AzAPI + Random providers; remote state backend in Azure Blob |
| `networks.tf` | VNet, VM subnet, MySQL delegated subnet, NSG, public IP, NIC |
| `vm.tf` | Ubuntu 22.04 LTS VM, SSH-only auth, User-Assigned Managed Identity, custom startup script |
| `db.tf` | MySQL Flexible Server in a private subnet, credentials from Key Vault |
| `keyvaults.tf` | Key Vault with RBAC auth, auto-generated secrets, scoped role assignments |
| `acr.tf` | Azure Container Registry, admin access disabled |
| `storage.tf` | Storage Account + private blob container |
| `data.tf` | Data sources: resource group, SSH key, Azure AD group, current client |
| `variables.tf` | All configurable values with sensible defaults |
| `outputs.tf` | MySQL FQDN, ACR login server, storage account name, VM public IP, Key Vault URI |

## Security decisions worth noting

**No hardcoded secrets.** MySQL credentials are generated by Terraform's `random_password` resource and stored directly in Key Vault. The database reads the password from Key Vault at provision time вЂ” it never appears in code.

**Key Vault uses RBAC, not access policies.** `enable_rbac_authorization = true` means permissions are managed through Azure role assignments rather than the older vault-level access policy model. This integrates cleanly with Azure AD and is easier to audit.

**ACR admin is disabled.** Container pulls require a managed identity or service principal вЂ” no shared username/password floating around.

**VM has no password authentication.** `disable_password_authentication = true` вЂ” SSH key only, with the public key sourced from an existing Azure SSH Key resource rather than hardcoded in Terraform.

**MySQL is subnet-delegated.** The database lives in a subnet delegated to `Microsoft.DBforMySQL/flexibleServers` with no public endpoint. Only resources inside the VNet can reach it.

**`depends_on` for Key Vault secret.** The secret creation explicitly waits for the Secrets Officer role assignment to complete before writing. Without this, Terraform races ahead and hits a permissions error intermittently.

## Prerequisites

- Terraform >= 1.3
- Azure CLI authenticated (`az login`)
- An existing Resource Group
- An existing SSH Public Key resource in Azure
- An existing Azure AD group for your DevOps team
- A User-Assigned Managed Identity (created separately or in a bootstrap step)
- A Storage Account for remote state (created separately to avoid chicken-and-egg)

## Usage

```bash
# Clone the repo
git clone https://github.com/psobeleagu/azure-infra-terraform
cd azure-infra-terraform

# Copy the example vars file and fill in your values
cp terraform.tfvars.example terraform.tfvars

# Initialise вЂ” pulls providers and connects to remote state backend
terraform init

# Review what will be created
terraform plan

# Apply
terraform apply
```

To deploy a second environment, create a separate `terraform.tfvars` with `environment = "staging"` and run:
```bash
terraform apply -var-file="staging.tfvars"
```

## Outputs

After `terraform apply`:

```bash
terraform output mysql_server_fqdn               # MySQL connection host
terraform output container_registry_login_server # ACR endpoint
terraform output storage_account_name            # Storage account name
terraform output load_balancer_public_ip         # VM public IP
terraform output key_vault_uri                   # Key Vault URI

# Sensitive вЂ” requires -json flag
terraform output -json mysql_admin_password
```

## What I'd improve for production

- Lock down the NSG SSH rule to specific IP ranges rather than `*` (currently open for dev convenience)
- Add an Azure Private DNS Zone for MySQL so the FQDN resolves correctly inside the VNet
- Tighten Key Vault network ACLs from `Allow` to `Deny` with an explicit subnet allowlist
- Add `azurerm_monitor_diagnostic_setting` to ship Key Vault audit logs and VM metrics to Log Analytics
- Move the Managed Identity provisioning into this same Terraform module rather than treating it as a manual prerequisite
